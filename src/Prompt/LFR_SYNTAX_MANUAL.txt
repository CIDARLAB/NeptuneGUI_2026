Neptune 2026 - LFR Syntax Manual (Quick Reference)

Canonical implementation lives in:
- Neptune_2026/docs/LFR_READABLE_SYNTAX_SPEC_V2.md
- Neptune_2026/pylfr/lfrX.g4

1) Basic module form
module <module_name>(<io_block>);
  <declarations and statements>
endmodule

2) IO declarations
- finput
- foutput
- control

Module port list (important):
module name(
    finput in_a, in_b,
    foutput out_1
);

Use commas between port groups inside parentheses.
Do not use semicolons between finput/foutput/control in the module header.
Invalid: module bad(finput B, C; foutput A;);

Vector form is supported, for example:
finput [0:15] inputs
control [0:3] route

3) Common declarations
- flow
- storage
- number
- signal
- pump

Examples:
flow a_to_mix, mix_out;
storage stage_1;
number ratio = 2, gain = 10;

4) Behavior statements
4.1 assign
assign <lhs> = <expression>;

Common operators used in Neptune examples:
- +   merge
- -   branch/waste style
- %   metering/sampling style
- /N  split style
- ~   mapped primitive style

4.2 distribute@(control)
Use distribute blocks for control-sensitive routing:

distribute@(sel)
begin
  if (sel == 1'b0) out <= a;
  else out <= b;
end

case(...) forms are also supported.

Current distribute condition scope (Neptune 2026):
- Supported operators in if/else if:
  == != && || & | ^ ^~ ~^
- Supports chained expressions and parentheses, for example:
  if (c1 && c2 || c3)
  if ((c1 && c2) || c3)
- Precedence (high to low):
  == !=, then &, then ^/^~/~^, then |, then &&, then ||
- Signal-vs-signal ops with == != & | ^ ^~ ~^ require matching widths.
- Prefer explicit parentheses when mixing logical and bitwise operators.

7) Cross-file imports (Neptune 2026 + NeptuneGUI Editor)
Verilog-style backtick imports are supported:
`import "Valve.lfr"
`import "library/Valve.lfr"
`import "/abs/path/Valve.lfr"

In the NeptuneGUI Editor, import another workspace's LFR module with:
`import "WorkspaceName/file.lfr"

where WorkspaceName is the Dashboard workspace name and file.lfr is an LFR
file stored in that workspace. Example:
`import "Example/flow_only_demo.lfr"

Before compile, the Editor extracts every `import path, checks that each file
exists (including transitive imports), and sends only those files to the
backend. Missing paths or circular imports stop compile with an error.

Rules:
- one import per line
- quoted path must end with .lfr
- transitive import is supported
- circular import raises an error (Editor check + preprocessor)

8) Authoring rules
- Keep source ASCII when possible.
- Drive declared outputs explicitly.
- Use <= inside distribute branches; use = in assign.
- Prefer explicit intermediate flow nodes for complex expressions.
